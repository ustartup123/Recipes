## Problem Statement

Aquarium hobbyists need to track water parameters, maintenance, and events across multiple tanks (freshwater and marine). ~10 existing apps serve this market (Aquarimate, AquaticLog, Aquarium Tracker) but none well: they're unpolished, add friction instead of removing it, and treat parameter tracking as the end product rather than a means to actionable insight.

The founder keeps tanks and uses "a mix of everything" — some notes, some memory, some apps — because no single tool has earned daily use. The missing piece is not more data entry; it's an AI layer that connects events (fish additions, deaths, dosing) to parameter trends and proactively tells you what to do before problems escalate.

## What Makes This Cool

The "whoa" moment: you open the app and it says "Your ammonia has risen 0.15ppm over the last 4 readings. This correlates with adding 4 neon tetras on Tuesday. Your tank is likely in a mini-cycle. Do a 25% water change tonight." Nobody does this. Every existing app shows you graphs and expects you to draw your own conclusions.

The event-to-parameter correlation is the core innovation. Parameters without context are just numbers. Events without parameters are just a journal. The AI connecting both is the product.

## Constraints

- Build on existing Next.js 14 / Firebase / Tailwind / Gemini stack
- Google auth already implemented
- Currently freshwater-only; marine support needed but not in this phase
- No LFS/B2B features in this phase — validate the core thesis first
- Must be useful for the founder's own tanks immediately

## Premises

1. **Existing apps add friction** — the problem is UX, not features. The founder uses "a mix of everything" instead of committing to one tool.
2. **The killer feature is proactive AI + event logging** — not data entry. The AI needs event context to give actionable advice. (Challenged by second opinion — see below — but founder defended this premise.)
3. **LFS white-label is the distribution play** — stores hand app to customers, product recs drive revenue, store gets customer stickiness. (Deferred to future phase.)
4. **Marine support is must-have** — reef keepers are power users who track the most, spend the most, and will pull their LFS into adopting this. (Deferred to next phase after events.)

## Cross-Model Perspective

An independent Claude subagent reviewed the session cold. Key findings:

- **Steelman:** "Shopify-meets-Toast for the aquarium trade. You own the point of insight (the moment a hobbyist needs something) and connect it to the point of sale." The data flywheel — more users logging → better AI → more trust → more product conversions — is the moat.
- **Most revealing quote:** "Local fish stores would love to white label it and provide it to their customers and sell product through there." The subagent noted this reveals B2B2C thinking, not consumer app thinking. The real customer may be the store owner.
- **Challenged premise:** Argued proactive AI should come AFTER zero-friction capture (photo-snap test strips). "The ones that win (Apple Health, Dexcom) win on zero-friction capture, not on insights. The insights come free once you solve capture." Founder defended the original premise.
- **48-hour prototype:** Strip to one screen with four big buttons (Death, Addition, Dose, Water Change), add test strip photo upload via Gemini vision, build a "store owner" demo view with seeded data.

## Approaches Considered

### Approach A: "Smart Journal" (Chosen)
Add event logging + AI correlation to the existing app. Events (fish death, addition, dosing, water change notes) with timestamps. AI advisor gets event context. Add marine parameters to schema. No LFS features yet.
- Effort: S (human: ~3 days / CC: ~30 min)
- Risk: Low
- Pros: Ships fast, immediately useful, validates core thesis
- Cons: No LFS angle, no photo input, no multi-language

### Approach B: "Store-Ready MVP"
Full event system + marine params + i18n + product inventory + basic LFS dashboard + photo test strip input.
- Effort: M (human: ~2 weeks / CC: ~2 hours)
- Risk: Medium (scope creep, LFS assumptions untested)
- Pros: Complete enough to demo to a fish store
- Cons: Building LFS features before talking to a single store owner

### Approach C: "One Fish Store"
Build only what's needed to demo to one local fish store. Event logger, photo input, marine params, fake store owner view with seeded data.
- Effort: S (human: ~2 days / CC: ~30 min)
- Risk: Low
- Pros: Tests LFS thesis with real evidence
- Cons: Throwaway code, no production path

## Recommended Approach

**Approach A: Smart Journal.** Build the core value first, use it yourself, prove the AI correlation thesis works. The LFS angle is exciting but premature — validate the product before building the platform.

## Implementation Spec

### 1. Unified Event Data Model

**Core principle: Everything is an event.** Water parameter readings, water changes, fish additions, deaths, dosing... all stored in one `events` collection. This gives a single unified timeline and simplifies the data model.

New Firestore collection: `events`

**Firestore security rules:** Events are scoped to the authenticated user.
- Create: `allow create: if request.auth != null && request.auth.uid == request.resource.data.userId;`
- Read/update/delete: `allow read, update, delete: if request.auth != null && request.auth.uid == resource.data.userId;`
- **Composite index required:** `events` collection — `userId` ASC + `aquariumId` ASC + `timestamp` DESC.

```typescript
type EventType =
  | 'parameter_reading'
  | 'water_change'
  | 'fish_added'
  | 'fish_died'
  | 'dosing'
  | 'equipment'
  | 'observation'
  | 'other';

interface AquariumEvent {
  id: string;
  aquariumId: string;
  userId: string;
  timestamp: Timestamp;
  type: EventType;
  title: string;           // auto-generated or user-provided, e.g., "Water Test", "Added 4 Neon Tetras"
  details?: string;        // free-text notes
  metadata?: {
    // Type-specific structured data
    // parameter_reading
    ph?: number;
    ammonia?: number;       // ppm
    nitrite?: number;       // ppm
    nitrate?: number;       // ppm
    temperature?: number;   // Fahrenheit internally
    gh?: number;            // dGH
    kh?: number;            // dKH
    // fish_added / fish_died
    species?: string;
    count?: number;
    source?: string;        // for fish_added (e.g., "PetSmart")
    // dosing
    product?: string;       // e.g., "Seachem Prime"
    amount?: string;        // e.g., "2ml"
    // water_change
    waterChangePercent?: number;
    conditionerUsed?: string;
    // Marine params (forward-looking, phase 2)
    salinity?: number;      // ppt
    calcium?: number;       // ppm
    alkalinity?: number;    // dKH
    magnesium?: number;     // ppm
    phosphate?: number;     // ppm
  };
  aiCorrelation?: string;  // AI-generated badge text, e.g., "Linked to ammonia spike"
}
```

**Event type → icon + color mapping:**

| Type | Icon | Color | Hex |
|------|------|-------|-----|
| `parameter_reading` | FlaskConical (Lucide) | Blue | `#3b82f6` / `blue-500` |
| `water_change` | Droplets (Lucide) | Teal | `#14b8a6` / `teal-500` |
| `fish_added` | Fish (Lucide) | Green | `#22c55e` / `green-500` |
| `fish_died` | Skull (Lucide) | Red | `#ef4444` / `red-500` |
| `dosing` | Pill (Lucide) | Purple | `#a855f7` / `purple-500` |
| `equipment` | Wrench (Lucide) | Amber | `#f59e0b` / `amber-500` |
| `observation` | Eye (Lucide) | Slate | `#64748b` / `slate-500` |
| `other` | MoreHorizontal (Lucide) | Slate | `#64748b` / `slate-500` |

Each icon renders at 10% opacity bg, 20% border, full-color text (same pattern as existing badges).

**Migration from existing collections:**

Fresh start. No migration needed. The existing `parameters` and `waterChanges` collections will be dropped. All new data goes to `events`. The old `/parameters` and `/water-changes` pages will be deleted (not deprecated).

**Validation rules** (enforced client-side before write):

| Field | Valid range |
|-------|-------------|
| pH | 0 – 14 |
| Ammonia | 0 – 20 ppm |
| Nitrite | 0 – 20 ppm |
| Nitrate | 0 – 500 ppm |
| Temperature | 32 – 120 °F |
| GH | 0 – 30 dGH |
| KH | 0 – 30 dKH |
| count (fish) | 1 – 999 |
| waterChangePercent | 1 – 100 |

**Key design decisions:**
- Event type is the FIRST choice (not buried in a dropdown below parameters). Pill-style selector at the top, horizontally scrollable on mobile. "Test" (parameter_reading) is the default selection.
- Each event type reveals its own fields. No combined form with everything visible.
- Placeholders for parameter fields show last reading's values (fetched from most recent `parameter_reading` event).
- Notes field is always visible below event-specific fields.

- URL accepts `?type=fish_added` (etc.) to pre-select event type from Quick-Add buttons.
- Events can be submitted without parameters. Parameter readings can be submitted without other events. This is not a combined form. It's an event logger where "water test" is one event type.

**Severity banner component:**

| Severity | Color | Icon | Label | When |
|----------|-------|------|-------|------|
| Green | emerald-500 bg/border | CheckCircle | ALL GOOD | All params in range, stable trends |
| Yellow | yellow-500 bg/border | AlertTriangle | MINOR CONCERN | Slight drift, worth watching |
| Orange | orange-500 bg/border | AlertTriangle | ACTION RECOMMENDED | Clear issue, needs attention |
| Red | red-500 bg/border | XCircle | URGENT | Immediate danger to livestock |

Each severity level uses icon + text label + color (not color alone), for accessibility.

**Action buttons on insight card:**
- When the AI recommends a water change → show "Log Water Change" button (links to `/log?type=water_change`)
- When the AI recommends retesting → show "Log Retest" button (links to `/log?type=parameter_reading`)
- These buttons connect insight to action. The user reads the recommendation and can act on it in one tap.

**Stale analysis indicator:**
- If new events have been logged since the last analysis, show a subtle badge: "New data since last analysis" with a refresh icon next to the "Run New Analysis" button.

**Analysis history:**
- Store each analysis in Firestore: `analyses` collection with `aquariumId`, `userId`, `timestamp`, `severity`, `summary`, `recommendations`, `rawResponse`, `contextSnapshot` (what data went in).
- Show previous analyses in a collapsed accordion below the latest. Tap to expand.

**Loading state:**
- Shimmer skeleton matching the insight card layout. "Analyzing your tank..." text with animated dots.

**Rate limit state:**
- If 24 daily analyses reached: "You've hit today's analysis limit (24). Try again tomorrow." Friendly, not punitive. Show the existing latest analysis.

### 4. AI Correlation Badge Generation

**When:** Badges are generated during on-demand analysis runs (Section 3D). When the user taps "Run Analysis," the API:
1. Fetches last 30 days of events + last 20 parameter readings
2. Sends to Gemini with a prompt requesting both the analysis AND correlation badges
3. Receives structured JSON response with analysis + an array of `{ eventId, badgeText }` pairs
4. Writes the analysis to the `analyses` collection
5. Writes `aiCorrelation` badges back to the relevant events in the `events` collection

**Prompt structure for badge generation:**
```
In addition to the analysis, identify any events that correlate with parameter changes.
For each correlation, return:
- eventId: the event's ID
- badgeText: a short (under 60 chars) description of the correlation
  e.g., "Ammonia started rising 2 days after this event"
  e.g., "Ammonia dropped 0.10ppm after this change"
  e.g., "Parameters were stable — possible stress from new additions"
```

**When no events exist (only parameter readings):**
The analysis prompt includes a directive: "This tank has parameter readings but no logged events (fish additions, water changes, dosing, etc.). Provide a range-based analysis of current values and trends. Note that event logging would enable correlation insights. Do not fabricate correlations."

**When everything is stable (green result):**
The prompt includes: "For healthy/stable tanks, highlight specific positive observations: stability duration, improving trends, parameter consistency. Avoid generic 'everything looks good' responses. Example: 'Your pH has held steady at 7.2 for 3 weeks, that's excellent stability. Nitrates are trending down slightly, suggesting your water change schedule is dialed in.'"

## Design Review & Feedback

### What's Working

**The dark theme is strong.** Slate-950 background with teal-500 accent is high-contrast, easy on the eyes, and has aquarium-appropriate vibes. The gradient accent line on cards (`h-0.5 bg-gradient-to-r from-teal-500/40`) is a nice subtle touch.

**The badge system is well-designed.** The 10% opacity backgrounds with 20% borders (`.badge-safe`, `.badge-warning`, etc.) are the right pattern. Readable without being loud.

**The parameter grid on dashboard cards is information-dense without being cluttered.** Seven params in a responsive grid with sparklines... that's a lot of data in a small space and it reads cleanly.

### Design Feedback: Toward Apple-Like Simplicity

Apple's design language is about restraint: fewer elements, more whitespace, larger type, fewer colors, zero decoration that doesn't earn its keep. Here's where AquaTrack can move in that direction.

#### 1. Typography: Fira Code headings feel "developer tool," not "consumer app"

Fira Code for headings gives AquaTrack a terminal/hacker aesthetic. That's cool for a dev tool but works against the "simple and smart" goal. Apple uses one sans-serif family at different weights.

**Recommendation:** Use Fira Sans for everything (already loaded). Use weight contrast instead of font-family contrast: 700 for headings, 400 for body, 300 for secondary text. Keep Fira Code only for parameter values (numbers in monospace make sense for readability).

This is a small CSS change:
```css
h1, h2, h3, h4, h5, h6 {
  font-family: var(--font-fira-sans), sans-serif;
  font-weight: 700;
}
.mono-value {
  font-family: var(--font-fira-code), monospace;
}
```

#### 2. Color palette: use fewer colors, more deliberately

Currently the app uses teal (accent), blue (freshwater badge), emerald (planted badge, safe), amber (brackish badge), yellow (warning), red (danger), plus the new event type colors (green, purple, etc.). That's a lot of color.

**Recommendation:** Simplify to 3 communication colors + 1 accent:
- **Teal** — primary accent, interactive elements, positive/brand
- **Yellow/Amber** — caution, warnings
- **Red** — danger, errors
- **Slate** — everything else (text, borders, backgrounds, neutral states)

Event type colors are fine because they serve information-density on the timeline. But badges, buttons, and decorative elements should stick to the 3+1 system. Remove blue as a distinct color (use teal instead for freshwater).

#### 3. Whitespace: the dashboard is dense

Apple's apps breathe. The current dashboard packs stats, tank cards (with 7-param grids), quick actions, and footer rows tightly. On mobile this creates a wall of information.

**Recommendation:**
- Increase card padding from `p-5` to `p-6`
- Increase gap between tank cards from `space-y-4` to `space-y-6`
- The parameter grid is great on desktop (7 columns). On mobile, show only the 3-4 params that matter most (or that have alerts) with a "See all" expandable.
- Remove the summary stat row on mobile. It takes vertical space for information the tank cards already show.

#### 4. Reduce element count per screen

Apple's philosophy: if you can remove an element without losing meaning, remove it.

Current dashboard card has: type badge + volume + alert badge + tank name + 7 param cells + sparklines + last reading date + last change date + Ask AI link + 3 action buttons. That's ~15-20 distinct elements per card.

**Recommendation for tank card simplification:**
- Lead with tank name (largest, boldest)
- Show alert status inline with name (not a separate badge)
- Show the 3 most relevant parameters (the ones with alerts, or pH + ammonia + nitrate as defaults)
- "See all parameters" link for the rest
- Recent events (2-3 lines)
- 2 action buttons max on mobile (Log + Timeline). "Analyze" and "Ask AI" accessible via the Timeline page.

#### 5. Animations: less is more

The shimmer and glow-pulse animations are fine for loading and danger states. But avoid adding entrance animations, slide-ins, or transitions for the new event timeline. Apple's animations are purposeful (they communicate state changes) and fast (200-300ms). The existing `transition-all duration-150` is good.

#### 6. The Quick-Add bar is the most Apple-like element in the design

Four or five large, obvious, one-tap buttons for the most common actions. This is exactly how Apple designs primary interactions (think the Camera app's mode selector or the Health app's log buttons). Keep this design. Make the icons clean Lucide icons (not emoji). Make the labels short (one word). Make the touch target large (48x48px minimum).

## Deployment & Security

#### Public Deployment

The app will be deployed as a public website (e.g., via Vercel or GCP Cloud Run) so the founder can use it on mobile immediately. Mobile-first responsive design is already in place.

#### Data Isolation

Every Firestore query MUST be scoped by `userId`. Firestore security rules enforce this at the database level (see Section 1). No user should ever see another user's aquariums, events, or parameter data. This applies to:
- All client-side queries (filter by `userId`)
- All security rules (read/write gated on `request.auth.uid == resource.data.userId`)
- The AI advisor endpoint (only passes the authenticated user's data to Gemini)

#### Gemini API Key Security

The Gemini API key is used across multiple apps. Security requirements:
- **Never expose the key client-side.** All Gemini calls go through the Next.js API route (`/api/advisor`, `/api/analysis`). The key lives in environment variables on the server only.
- **Store in `.env`** locally, use **GCP Secret Manager** for production deployment (never pass as `--set-env-vars` in deploy commands).
- `.env` must be in `.gitignore`. Provide `.env.example` with placeholder values.
- **Startup check:** The API routes must fail loudly if `GEMINI_API_KEY` is not set.

#### Vertex AI SDK + Labels

Switch from `@google/generative-ai` to `@google-cloud/vertexai`. Vertex AI SDK uses service account auth (same as Firebase Admin SDK, no separate API key). Every Vertex AI call includes labels for billing attribution:
- `app: "aquatrack"` — identifies which app made the call
- `user_id: <firebase_uid>` — identifies which user triggered it
- Vertex AI labels are native to the SDK and show up in GCP billing dashboards
- Use Gemini structured output mode for analysis responses (reliable JSON, no parsing failures)
